# afeaturemerge

参考一个系统的功能，系统性地分析并规划如何在我们自己的系统中实现，产出可执行的需求文档和技术方案。

---

## 第一步：信息收集

**先解析 → 再自补 → 确认后动手**

### 清单（按序完成）

1. **解析用户输入**——从消息中提取已知信息
2. **自行补全**——在追问前，先尝试从代码库/网络获取缺失信息
3. **展示理解摘要**——向用户确认理解，收到确认后才开始调研

<HARD-GATE>
未展示摘要并收到用户确认前，不得开始调研。
</HARD-GATE>

### 解析清单

从用户消息中识别以下信息：

| 信息项 | 若缺失，如何处理 |
|-------|--------------|
| 参考系统名称及目标功能 | **必须追问**，无法继续 |
| 我方目标应用/模块 | **必须追问**，无法继续 |
| 参考系统文档/代码位置 | 自行 `WebSearch`，无需追问 |
| 我方代码位置 | 自行 `Grep` 搜索，无需追问 |
| 特殊需求/约束 | 无则默认对齐参考系统 |

### 理解摘要模板

解析完成后，向用户发送：

> 我的理解：
> - 参考系统：**[X 系统的 Y 功能]**
> - 目标应用：**[Z 应用/模块]**
> - 实现目标：**[具体能力描述]**
> - 特殊约束：**[无 / 列出约束]**
>
> 确认后我开始调研，或告诉我需要调整的地方。

### 追问规则

若有关键信息缺失（参考系统或目标应用不明确）：

- **每次只问一个问题**，不要列出多个问题
- **优先用选项**（选择比填空更高效）
- 收到回答后更新摘要，再次确认

> **示例**：目标应用不明确时
> "你希望在哪个应用里实现这个功能？
> A) [应用 A 名称]
> B) [应用 B 名称]
> C) 其他（请告诉我）"

**没有明确说明的地方，默认与参考系统保持一致。**
若无参考系统可对标，直接进入 `brainstorming` 进行分析。

---

## 第二步：并行调研

两路调研独立进行，同时开展。

### A. 参考系统调研

**目标**：深入理解参考系统的功能实现。

| 资源类型 | 工具 |
|---------|------|
| 公网文档 | `WebFetch` / `WebSearch` |
| GitHub 源码 | `WebFetch`（raw.githubusercontent.com） |
| 内部知识库 | 内部知识库 MCP（如已配置） |
| 用户提供的链接 | 文档获取 MCP 或 `WebFetch` |
| 用户提供的代码位置 | `Glob` / `Grep` / `Read` |

**调研重点**：
- 核心设计理念（为什么这样设计）
- 关键接口/API 及调用方式
- 典型使用示例（代码片段）
- 已知限制或注意事项

### B. 现有系统调研

**目标**：摸清我们当前的实现状态。

若**无现有实现**，直接记录"无现有实现"，跳过本步骤。

1. 用 `Glob` 查找相关目录和文件
2. 用 `Grep` 搜索核心类名、方法名、注解
3. 用 `Read` 阅读核心实现
4. 若有内部文档，用内部知识库 MCP（如已配置）检索

**调研重点**：
- 目前是否有类似功能？实现程度如何？
- 核心类/接口在哪里？
- 现有实现的局限性是什么？

---

## 第三步：产出分析文档

基于调研结果，产出三份结构化文档。

### 文档一：参考系统功能实现分析

```markdown
# [参考系统名] - [功能名] 实现分析

> 分析日期：YYYY-MM-DD
> 参考系统：[名称及版本]
> 文档/代码来源：[链接]

## 功能概述

（核心设计理念，3-5句话）

## 核心实现机制

（关键技术点，说明它如何工作）

## 关键接口 / API

（最重要的接口，带入参/出参说明）

```代码示例```

## 典型使用方式

（开发者如何使用，代码示例）

## 已知限制

（注意事项、边界条件）
```

### 文档二：现有系统功能情况分析

```markdown
# [我们的系统] - [功能名] 现状分析

> 分析日期：YYYY-MM-DD
> 系统：[应用/模块名称]

## 现有能力

（目前支持什么，哪怕部分支持也列出）
若无现有实现，直接写：**当前无相关实现**

## 当前实现

（核心实现位置：包名/类名/文件路径）

```代码示例（核心片段）```

## 已知局限

（现有实现的不足之处）
```

### 文档三：如何在现有系统中实现参考系统类似的功能

这是核心文档，分两部分：

#### 3.1 对比分析表

| 功能点 | 参考系统 | 我们系统 | 是否可参考 | 说明 |
|--------|---------|---------|-----------|------|
| 功能 A | ✅ 支持 | ❌ 无 | ✅ 可直接参考 | 实现思路相同 |
| 功能 B | ✅ 支持 | ⚠️ 部分 | ✅ 可参考改造 | 需适配我们的框架 |
| 功能 C | ✅ 支持 | ❌ 无 | ❌ 不可参考 | 依赖其闭源组件 |

#### 3.2 可以参考的部分

列出可以参考实现的功能点，**调用 `brainstorming` 进行分析**，产出：

- **需求文档**：明确要实现什么，用户价值，验收标准
- **技术方案**：具体实现步骤，涉及的类/模块，代码改动范围

#### 3.3 不可以参考的部分

列出无法直接参考的功能点（依赖专有实现、技术栈不兼容等），**调用 `brainstorming` 进行分析**，从头设计，产出：

- **需求文档**：明确要实现什么，用户价值，验收标准
- **技术方案**：独立设计的实现方案

## 第四步：同步文档

三份分析文档完成后，按以下优先级决定同步方式：

```
1. 是否有文档同步 skill（检查当前可用 skill 列表）？
   → 有：调用该 skill，将三份文档同步到对应知识库目录

2. 是否有文档创建 MCP 工具（如钉钉/语雀/Confluence 等）？
   → 有：直接调用 MCP 工具创建文档

3. 以上都没有：
   → 在当前工作目录下创建 docs/ 文件夹（如不存在），
     写入三个本地 Markdown 文件：
     - docs/01-ref-system-analysis.md
     - docs/02-current-system-analysis.md
     - docs/03-implementation-plan.md
     并告知用户文件已保存到本地。
```

---

## brainstorming 调用说明

对"可参考"和"不可参考"的每个功能块，分别调用 `brainstorming`。

**调用时提供的上下文**：
- 参考系统的相关实现（文档三·3.1 的分析结论）
- 我们系统的现有状态
- 需求的特殊要求（无特殊要求则默认对齐参考系统）
- 明确告知：可参考 → "请基于参考系统的实现方式给出方案"；不可参考 → "请独立设计方案"

---

## 调研质量标准

- **有依据，不猜测**：结论来自实际文档或代码
- **两边都深入**：参考系统和我们系统都要认真研究
- **差距有证据**：指出"我们缺 X"时，要有参考系统的来源和我们缺失的代码证据
- **方案可落地**：实现路径具体到模块/类级别
- **代码片段精选**：只保留最能说明问题的关键片段

---

## 资源不足时的处理

| 情况 | 处理方式 |
|------|---------|
| 参考系统文档找不到 | `WebSearch` 搜索，或请用户提供链接 |
| 参考系统是闭源/内部系统 | 说明情况，请用户提供文档或描述 |
| 我们系统代码范围不确定 | 先用 `Grep` 搜关键词，逐步缩小范围 |
| 无现有实现 | 记录"无现有实现"，文档三的分析全部走"不可参考"流程（即 brainstorming 独立设计） |
| 两边差距极小 | 诚实说明，指出可直接复用的具体实现 |

## 示例触发

**示例 1（有现有实现）**：
> 「参考 LangChain 的 Memory 机制，在 my-chat-app 里实现会话记忆功能」

执行顺序：
1. 确认信息完整（参考系统、目标系统、功能点明确）
2. 并行：调研 LangChain Memory 文档 + 搜索 my-chat-app 现有会话存储实现
3. 产出文档一（LangChain Memory 分析）、文档二（现有实现分析）
4. 产出文档三，区分可参考/不可参考部分，分别调用 brainstorming
5. 同步文档（按第四步优先级决定方式）

**示例 2（无现有实现）**：
> 「参考 OpenAI Assistants API 的 Thread 设计，给我们的 API 加多轮对话支持」

执行顺序：
1. 追问：目标系统是哪个模块？有无特殊要求？
2. 并行：调研 OpenAI Assistants API 文档 + 确认我方无现有实现
3. 产出文档一（OpenAI Thread 分析）、文档二（当前无相关实现）
4. 文档三全部走"不可参考"流程，调用 brainstorming 独立设计
5. 同步文档
