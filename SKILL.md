  - 用户说"参考 X 系统/框架的 Y 功能，在我们系统里实现"
  - 用户说"调研 X 是怎么做 Y 的，然后给出我们的实现方案"

# afeaturemerge

参考一个系统的功能，系统性地分析并规划如何在我们自己的系统中实现，产出可执行的需求文档和技术方案。

---

## 第一步：信息收集

**先检测环境 → 解析输入 → 再自补 → 确认后动手**

### 清单（按序完成）

1. **检测文档同步配置**——判断本地是否具备云端同步能力，确定文档落地方式
2. **解析用户输入**——从消息中提取已知信息
3. **自行补全**——在追问前，先尝试从代码库/网络获取缺失信息
4. **展示理解摘要**——向用户确认理解，收到确认后才开始调研

<HARD-GATE>
未完成同步配置检测、未展示摘要并收到用户确认前，不得开始调研。
</HARD-GATE>

### 文档同步配置检测

**在解析需求之前**，先检测当前会话中是否具备云端知识库同步能力：

**检测方式**（两项均检查）：
- 当前会话可用 skill 列表中是否包含文档同步类 skill（名称中含 `sync`、`doc`、`kb`、`knowledge` 等关键词）
- 可用 MCP 工具中是否包含知识库写入类工具（能创建/更新文档的 MCP）

**根据检测结果执行不同分支**：

#### 分支 A：未检测到任何同步能力

直接告知用户：「未检测到云端知识库同步配置，过程文档将仅保存到本地 `docs/` 目录。」

记录 `sync_mode = LOCAL`，继续执行后续步骤，无需追问。

#### 分支 B：检测到同步能力

使用 `AskUserQuestion` 让用户选择文档落地方式：

```
AskUserQuestion({
  questions: [{
    header: "文档落地方式",
    question: "检测到云端知识库同步配置，过程文档保存到哪里？",
    multiSelect: false,
    options: [
      { label: "仅本地", description: "文档保存到本地 docs/ 目录，不同步云端" },
      { label: "仅云端知识库", description: "文档同步到云端知识库，不保存本地" },
      { label: "本地 + 云端知识库", description: "同时保存本地并同步云端（推荐）" }
    ]
  }]
})
```

根据用户选择记录 `sync_mode`：
- 仅本地 → `sync_mode = LOCAL`
- 仅云端 → `sync_mode = CLOUD`
- 本地 + 云端 → `sync_mode = BOTH`

### 收到用户确认后，立即建档（调研开始前完成）

用户确认需求信息正确后，**在开始调研之前**，根据 `sync_mode` 执行建档：

**若 `sync_mode` 包含 CLOUD**（即 CLOUD 或 BOTH）：

1. 调用已配置的文档同步 skill/MCP 的建档流程，创建任务文件夹，记录返回的文件夹 ID 供后续使用
2. 在文件夹下创建「任务概述」文档，内容包含：
   - 用户原始请求
   - 任务目标（当前已知的）
   - 产出文档（留空，完成后补充）
   - 关键决策（留空，完成后补充）
3. 通知用户：「已创建云端任务文件夹，过程文档将实时同步。」

**若 `sync_mode = LOCAL`**：

跳过建档步骤，直接进入调研。

<HARD-GATE>
sync_mode 包含 CLOUD 时，必须在开始调研前完成建档，不得推迟到流程末尾。
</HARD-GATE>

### 解析清单

从用户消息中识别以下信息：

| 信息项 | 若缺失，如何处理 |
|-------|--------------|
| 参考系统名称及目标功能 | **必须追问**，无法继续 |
| 我方目标应用/模块 | **必须追问**，无法继续 |
| 参考系统文档/代码位置 | 自行 `WebSearch`，无需追问 |
| 我方代码位置 | 自行 `Grep` 搜索，无需追问 |
| 特殊需求/约束 | 无则默认对齐参考系统 |

### 理解摘要模板

解析完成后，先在消息中展示理解摘要，**然后紧跟一个 `AskUserQuestion` 确认工具调用**，让用户可以直接选择"确认，开始调研"或"需要调整"而无需手动输入：

> 我的理解：
> - 参考系统：**[X 系统的 Y 功能]**
> - 目标应用：**[Z 应用/模块]**
> - 实现目标：**[具体能力描述]**
> - 特殊约束：**[无 / 列出约束]**

```
AskUserQuestion({
  questions: [{
    header: "确认信息",
    question: "以上理解是否正确？",
    multiSelect: false,
    options: [
      { label: "确认，开始调研", description: "信息无误，可以继续" },
      { label: "需要调整", description: "请在 Other 文本框中说明需要修改的地方" }
    ]
  }]
})
```

### 追问规则

若有关键信息缺失（参考系统或目标应用不明确）：

- **每次只问一个问题**，不要列出多个问题
- **必须使用 `AskUserQuestion` 工具呈现选项**，不要在消息文本中写 A/B/C 列表——工具提供可用光标/回车选中的交互 UI，用户体验更好
- 收到回答后更新摘要，再次确认

**`AskUserQuestion` 使用规范**：
- `header`：问题的短标签（≤12 字），如"目标应用"、"功能范围"
- `question`：完整问题文字
- `options`：列出已知的候选项，**最后一项留给"其他"**，并在 `description` 里提示"请在 Other 文本框中填写"
- 若候选项超过 4 个，优先列出最可能的 3 项 + "其他"

**示例**（目标应用不明确时）：
```
AskUserQuestion({
  questions: [{
    header: "目标应用",
    question: "你希望在哪个应用里实现这个功能？",
    multiSelect: false,
    options: [
      { label: "应用 A", description: "..." },
      { label: "应用 B", description: "..." },
      { label: "其他", description: "请在 Other 文本框中填写应用名称" }
    ]
  }]
})
```

**没有明确说明的地方，默认与参考系统保持一致。**
若无参考系统可对标，直接进入 `brainstorming` 进行分析。

---

## 第二步：并行调研

两路调研独立进行，同时开展。

### A. 参考系统调研

**目标**：深入理解参考系统的功能实现。

| 资源类型 | 工具 |
|---------|------|
| 公网文档 | `WebFetch` / `WebSearch` |
| GitHub 源码 | `WebFetch`（raw.githubusercontent.com） |
| 内部知识库 | 内部知识库 MCP（如已配置） |
| 用户提供的链接 | 文档获取 MCP 或 `WebFetch` |
| 用户提供的代码位置 | `Glob` / `Grep` / `Read` |

**调研重点**：
- 核心设计理念（为什么这样设计）
- 关键接口/API 及调用方式
- 典型使用示例（代码片段）
- 已知限制或注意事项

### B. 现有系统调研

**目标**：摸清我们当前的实现状态。

若**无现有实现**，直接记录"无现有实现"，跳过本步骤。

1. 用 `Glob` 查找相关目录和文件
2. 用 `Grep` 搜索核心类名、方法名、注解
3. 用 `Read` 阅读核心实现
4. 若有内部文档，用内部知识库 MCP（如已配置）检索

**调研重点**：
- 目前是否有类似功能？实现程度如何？
- 核心类/接口在哪里？
- 现有实现的局限性是什么？

---

## 第三步：产出分析文档（产出即同步）

基于调研结果，产出三份结构化文档。**每份文档完成后立即**，按 `sync_mode` 执行：

| sync_mode | 本地保存 | 云端同步 |
|-----------|---------|---------|
| LOCAL | ✅ 保存到 `docs/` | ❌ |
| CLOUD | ❌ | ✅ 同步到任务文件夹 |
| BOTH | ✅ 保存到 `docs/` | ✅ 同步到任务文件夹 |

### 文档一：参考系统功能实现分析

**本地路径**：`docs/afeaturemerge/YYYY-MM-DD-[功能名]-ref-analysis.md`

**文档名**：`参考系统分析 — [参考系统名]`

```markdown
# [参考系统名] - [功能名] 实现分析

> 输入命令：[用户原始请求]
> 分析日期：YYYY-MM-DD

> 文档/代码来源：[链接]

## 功能概述

## 核心实现机制

（关键技术点，说明它如何工作）

## 关键接口 / API

（最重要的接口，带入参/出参说明）

```代码示例```

## 典型使用方式

（开发者如何使用，代码示例）

（注意事项、边界条件）
```

### 文档二：现有系统功能情况分析

**本地路径**：`docs/afeaturemerge/YYYY-MM-DD-[功能名]-current-analysis.md`

**文档名**：`现有系统分析 — [目标系统名]`

```markdown

若无现有实现，直接写：**当前无相关实现**

## 当前实现

（核心实现位置：包名/类名/文件路径）

```代码示例（核心片段）```

## 已知局限

（现有实现的不足之处）
```

### 文档三：对比分析与实现规划

**本地路径**：`docs/afeaturemerge/YYYY-MM-DD-[功能名]-implementation-plan.md`

**文档名**：`对比分析与实现规划`

这是核心文档，分两部分：

#### 3.1 对比分析表

| 功能点 | 参考系统 | 我们系统 | 是否可参考 | 说明 |
|--------|---------|---------|-----------|------|
| 功能 A | ✅ 支持 | ❌ 无 | ✅ 可直接参考 | 实现思路相同 |
| 功能 B | ✅ 支持 | ⚠️ 部分 | ✅ 可参考改造 | 需适配我们的框架 |
| 功能 C | ✅ 支持 | ❌ 无 | ❌ 不可参考 | 依赖其闭源组件 |

#### 3.2 可以参考的部分

列出可以参考实现的功能点，**调用 `brainstorming` 进行分析**，产出：

- **需求文档**：明确要实现什么，用户价值，验收标准
- **技术方案**：具体实现步骤，涉及的类/模块，代码改动范围

#### 3.3 不可以参考的部分

列出无法直接参考的功能点（依赖专有实现、技术栈不兼容等），**调用 `brainstorming` 进行分析**，从头设计，产出：

- **需求文档**：明确要实现什么，用户价值，验收标准
- **技术方案**：独立设计的实现方案

## 第四步：brainstorming → writing-plans（产出即同步）

### brainstorming 调用说明

对"可参考"和"不可参考"的每个功能块，分别调用 `brainstorming`。

**调用时提供的上下文**：
- 参考系统的相关实现（文档三·3.1 的分析结论）
- 我们系统的现有状态
- 需求的特殊要求（无特殊要求则默认对齐参考系统）
- 明确告知：可参考 → "请基于参考系统的实现方式给出方案"；不可参考 → "请独立设计方案"

**brainstorming 完成后（设计文档产出时）**，立即按 `sync_mode` 处理 `docs/superpowers/specs/` 下的设计文档（本地保存 / 云端同步 / 两者）。

**writing-plans 完成后（实现计划产出时）**，立即按 `sync_mode` 处理 `docs/superpowers/plans/` 下的实现计划（本地保存 / 云端同步 / 两者）。

### 完成时更新任务概述

所有文档产出后，若 `sync_mode` 包含 CLOUD，更新「任务概述」，补充：
- 产出文档清单（带链接）
- 关键决策摘要

---

## 调研质量标准

- **有依据，不猜测**：结论来自实际文档或代码
- **两边都深入**：参考系统和我们系统都要认真研究
- **差距有证据**：指出"我们缺 X"时，要有参考系统的来源和我们缺失的代码证据
- **方案可落地**：实现路径具体到模块/类级别
- **代码片段精选**：只保留最能说明问题的关键片段

---

## 资源不足时的处理

| 情况 | 处理方式 |
|------|---------|
| 参考系统文档找不到 | `WebSearch` 搜索，或请用户提供链接 |
| 参考系统是闭源/内部系统 | 说明情况，请用户提供文档或描述 |
| 我们系统代码范围不确定 | 先用 `Grep` 搜关键词，逐步缩小范围 |
| 无现有实现 | 记录"无现有实现"，文档三的分析全部走"不可参考"流程（即 brainstorming 独立设计） |
| 两边差距极小 | 诚实说明，指出可直接复用的具体实现 |

## 示例触发

**示例 1（有现有实现，已配置云端同步）**：
> 「参考 LangChain 的 Memory 机制，在 my-chat-app 里实现会话记忆功能」

执行顺序：
1. **检测同步配置**：发现已配置文档同步 skill，询问文档落地方式，用户选择"本地 + 云端知识库"→ `sync_mode = BOTH`
2. 解析需求，确认信息完整（参考系统、目标系统、功能点明确）
3. **收到用户确认后立即**：创建云端任务文件夹 + 「任务概述」，通知用户
4. 并行：调研 LangChain Memory 文档 + 搜索 my-chat-app 现有会话存储实现
5. 产出文档一（LangChain Memory 分析）→ **立即**按 BOTH 模式：保存本地 + 同步云端
6. 产出文档二（现有实现分析）→ **立即**按 BOTH 模式：保存本地 + 同步云端
7. 产出文档三（对比分析）→ **立即**按 BOTH 模式同步，区分可参考/不可参考部分，分别调用 brainstorming
8. brainstorming 产出设计文档 → **立即**按 BOTH 模式同步
9. writing-plans 产出实现计划 → **立即**按 BOTH 模式同步
10. 更新「任务概述」，补充产出清单和关键决策

**示例 2（无现有实现，未配置云端同步）**：
> 「参考 OpenAI Assistants API 的 Thread 设计，给我们的 API 加多轮对话支持」

执行顺序：
1. **检测同步配置**：未发现任何文档同步 skill/MCP，告知用户"仅保存本地"→ `sync_mode = LOCAL`
2. 追问：目标系统是哪个模块？有无特殊要求？
3. **收到用户确认后**：直接进入调研（无需建档）
4. 并行：调研 OpenAI Assistants API 文档 + 确认我方无现有实现
5. 产出三份分析文档 → **每份完成后立即**保存到本地 `docs/` 目录
6. 文档三全部走"不可参考"流程，调用 brainstorming 独立设计
7. brainstorming 产出设计文档 → **立即**保存本地
8. writing-plans 产出实现计划 → **立即**保存本地
